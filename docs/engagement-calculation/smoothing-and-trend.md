# Сглаживание и тренд engagement

Сырые engagement-значения `engagement_raw` зашумлены: отдельные флуктуации в эмоциях или кратковременные повороты головы могут давать скачки между соседними кадрами. Система применяет два независимых временных окна: одно для сглаживания итогового score, другое – для определения тренда.

Общая концепция сглаживания на всех уровнях пайплайна описана в [../pipeline/diff-level-smoothing.md](../pipeline/diff-level-smoothing.md). Здесь описывается только то, что касается уровня `EngagementCalculator`.

---

## Адаптивное окно сглаживания

Реализация: [`EngagementCalculator.calculate`](../../backend/app/services/video_processing/engagement_calculator.py#L279-L293).

### Параметры

| Параметр | Default | Смысл |
|----------|---------|-------|
| `window_size` | `45` | Максимум deque-окна `engagement_history` (~1.5 сек при 30 FPS) |
| `bypass_threshold` | `0.08` | Порог дисперсии, переключающий режимы |

### Логика

```python
engagement_history.append(engagement_raw)   # maxlen=45

if len(history) < 5:
    # прогрев
    smoothed = engagement_raw
else:
    recent = history[-15:]                  # последние ~0.5 сек
    if np.var(recent) < bypass_threshold:
        # стабильное состояние
        smoothed = np.mean(recent)          # короткое окно, быстрая реакция
    else:
        # изменчивое состояние
        smoothed = np.mean(history)         # полное окно до 45, сильное сглаживание
```

### Интерпретация режимов

| Режим | Условие | Окно | Характер отклика |
|-------|---------|------|------------------|
| Прогрев (стартовая фаза) | < 5 кадров в истории | – | Возвращается `engagement_raw` без изменений |
| Стабильный | `var(recent 15) < 0.08` | 15 кадров | Быстро сдвигается за реальным изменением |
| Изменчивый | `var(recent 15) ≥ 0.08` | до 45 кадров | Сглаживает резкие скачки |

> Нижний порог "прогрева" (стартовая фаза) в 5 кадров предотвращает выдачу сглаженного нуля в первые кадры сессии.


### Почему дисперсия по последним 15, а не по всему окну

`recent_window = history[-15:]` – при 30 FPS это окно ~0.5 сек. Достаточно длинное, чтобы зафиксировать локальную стабильность, и достаточно короткое, чтобы быстро переключаться в шумный режим при изменении поведения пользователя.

Использование дисперсии по всему `history` заставляло бы систему *застревать* в **изменчивом режиме** после любого единичного всплеска, что значительно замедлило бы отклик.


### Эмпирический смысл порога `0.08`

Стандартное отклонение `σ = 0.28` (`√0.08`). Это примерно размах `engagement_raw` при спокойной смене между `Normal` и `Alert`. При реальной смене из `Low` в `High` (перепад `0.3+`) дисперсия перескакивает порог и включается полное окно.

Значение `0.08` задано как default `bypass_threshold` в `EngagementCalculator.__init__`, может быть переопределено при создании экземпляра:

```python
EngagementCalculator(window_size=45, bypass_threshold=0.05, trend_window=30)
```

Env-переменных для этих параметров нет – настройка на данном этапе реализации только через код.

## Определение тренда

Реализация: [`EngagementCalculator._calculate_trend`](../../backend/app/services/video_processing/engagement_calculator.py#L336-L358).

### Параметры

| Параметр | Default | Смысл |
|----------|---------|-------|
| `trend_window` | `30` | Отдельное окно для тренда (~1 сек при 30 FPS) |
| `trend_threshold` | `0.05` (class-attribute `TREND_THRESHOLD`) | Порог разницы половин окна для `rising`/`falling` тренда |


### Алгоритм

```python
if len(trend_history) < 10:
    return "stable"

half = len(trend_history) // 2
first_half_mean  = mean(trend_history[:half])
second_half_mean = mean(trend_history[half:])
diff = second_half_mean - first_half_mean

if diff >  0.05: return "rising"
if diff < -0.05: return "falling"
return "stable"
```

### Логика

Сравниваются средние **первой и второй половины окна**. Это форма фильтра **low-pass**: случайная флуктуация в одном кадре размазывается по 15 точкам, её вклад в `diff` мал.

| Ситуация | `diff` | `trend` |
|----------|--------|---------|
| Пользователь начал улыбаться, score поднимается с 0.5 до 0.8 | `+0.3` | `rising` |
| Пользователь начал отвлекаться, score падает с 0.7 до 0.4 | `−0.3` | `falling` |
| Пользователь внимателен (без резких изменений поведения) | около `0` | `stable` |
| В истории < 10 кадров | N/A | `stable` |

### Почему не производная

Аппроксимация наклона регрессией (OLS на 30 точках) дала бы чуть более гладкий сигнал, но затратнее и чувствительнее к выбросам. Сравнение двух средних даёт грубо тот же результат при линейной сложности и защите от единичных шумовых кадров, что и лежит в основе данной реализации.

## Отношение сглаженного и сырого значения

В выдаче присутствуют **оба**:

```python
EngagementCalculateResult(
    score=round(engagement_smoothed, 3),
    score_raw=round(engagement_raw, 3),
    ...
)
```

- `score` – для отображения в UI, принятия решений, логирования. Устойчив к шуму.
- `score_raw` – для отладки и анализа мгновенных изменений. Полезен при построении графика "сырой vs сглаженный результаты" для визуальной верификации адаптивного окна.

API frontend'а по умолчанию использует `score` (см. [../backend/api-format.md](../backend/api-format.md)).

## Reset состояния

```python
engagement_calculator.reset()
# clears: engagement_history, trend_history, session_start_time, frame_count
```

Вызывается при старте новой сессии клиента. После reset первые 5 кадров вернут `score == score_raw` *(режим прогрева)*, первые 10 кадров – `trend == "stable"`.
