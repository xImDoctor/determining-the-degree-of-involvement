# Тонкости и ограничения фронтенда

В данной заметке представлен список того, что **не следует делать** с системой и почему. Важно отметить, что речь идёт не обагах, а конкретно о границах реализации, нарушение которых может приводить к крашу, лишним соединениям или некорректным результатам со стороны фронтенда.

---

## Общие для всех приложений

### 1. Не запускать второй экземпляр приложения с тем же `room_id` и `name`

`engagement_app.py` подключается к комнате `engagement-app` с именем `engagement-user` ([engagement_app.py:349](../../frontend/engagement_app.py#L349)). Если запустить два процесса streamlit одновременно, оба зайдут в одну комнату, у каждого будет свой `client_id` (UUID), но они будут конкурировать за вывод и логи.

Backend это не запрещает (т.к. все комнаты – multi-client), но в UI оба пользователя не увидят результаты друг друга. Используйте `tools/param_testing_app.py` для второго локального тестирования или поменяйте `room_id` в коде.

### 2. Не использовать одну веб-камеру в нескольких приложениях одновременно

Camera 0 эксклюзивна. Если запустить несколько приложений одновременно, и в обоих нажать "Запустить" – второе приложение получит `Не удалось открыть веб-камеру`. Запускать только одно за раз (на одном устройстве, для которого определена веб-камера).

### 3. Не оставлять приложение свернутым с открытой камерой

При свёрнутом окне Streamlit браузер может "задушить" таймеры, тогда health-check не будет срабатывать вовремя, метрики могут не обновляться. Камера при этом остаётся захваченной. Перед паузой используйте кнопку "⏹️ Стоп".

### 4. Не закрывать вкладку браузера без нажатия "⏹️ Стоп"

Streamlit не запускает teardown-логику при закрытии вкладки. WS-соединение и `cv2.VideoCapture` будут висеть в процессе Streamlit до явного перезапуска или остановки (завершения процесса Streamlit). На backend это приводит к "зависшим" клиентам в Redis.

Поэтому если вкладка случайно закрыта, то перезапустите `streamlit run`, чтобы освободить ресурсы. Backend сам подчистит клиента после WebSocketDisconnect, но фронтенд-процесс будет помнить старое состояние, чтобы это исправить – нужен перезапуск.

## Специфичные для `engagement_app.py`

### 5. Не редактировать `EAR_THRESHOLD` в `.env` ожидая изменения backend-логики

Переменная `EAR_THRESHOLD` встречается и в backend, и в frontend `.env`, но **с разными ролями**:

- На backend это реальный порог детекции моргания (см. [../pipeline/eye-aspect-ratio.md](../pipeline/eye-aspect-ratio.md)).
- На frontend (в этом приложении) это **только** позиция красной пунктирной линии на графике EAR (визуальный индикатор порога)

### 6. Не менять настройки параметров системы при работе UI

`engagement_app.py` **не имеет** слайдеров для backend-параметров. Это сделано намеренно – приложение для **демонстрации/наблюдения**, не для тюнинга/калибровки. Все backend-параметры задаются через `.env` при запуске backend.

Для подбора параметров используйте `tools/param_testing_app.py`, см. [param-testing-tool.md](param-testing-tool.md).

### 7. Не использовать камеры с индексом, отличным от 0

Код жёстко подключается к `cv2.VideoCapture(0)`. Если у вас:
- виртуальная камера (OBS, ManyCam) определена первой,
- встроенная веб-камера с индексом 1,

приложение откроет не ту камеру. Параметризации нет, поэтому нужна правка кода или системная настройка порядка камер на уровне ОС.

### 8. Не разворачивать приложение публично без авторизации

`engagement_app.py` **не имеет** механизмов аутентификации, т.к. служит в первую очередь для демонстрации работы пайплайна, преимущественно – в локальном окружении. Любой, кто открыл `http://<host>:8501`, попадёт на интерфейс с активной камерой пользователя. Если разворачиваете в сети, то закрывайте reverse proxy (nginx + auth) или VPN-периметром.

## Специфичные для `tools/param_testing_app.py`

### 9. Не пытаться запустить без `backend/`

Утилита импортирует backend модули напрямую через `sys.path`. Если каталог `backend/` отсутствует или зависимости не установлены, то на старте появится предупреждение `Не удалось импортировать модуль бэкенда`, а дальнейшая работа инструмента будет невозможной.

### 10. Не менять `window_size` посреди тестирования

Слайдер `window_size` пересоздаёт `deque(maxlen=n)` в `EmotionRecognizer`, **сбрасывая историю**. Сразу после изменения первые 14 кадров работают с пустой/неполной историей и temporal smoothing не действует. Подождите ~1 секунду перед оценкой результата.

## Crash-сценарии и действия

### Crash-сценарии

| Действие | Симптом | Причина |
|----------|---------|---------|
| Удалить `frontend/.env`, запустить frontend | Default-значения подтянутся, но не будет ошибки | `python-dotenv` игнорирует отсутствие файла |
| Указать невалидный `BACKEND_WS_URL` (например, `ftp://...`) | Ошибка подключения, кнопка камеры не активируется (через health-check) | `EngagementAPIClient.connect` бросит `ConnectionError` |
| Запустить utility вне `frontend/` | `sys.path.append` посчитает неверный путь, импорт упадёт | Путь backend строится относительно `__file__` – нужно из `frontend/tools/` |
| Удалить `frontend/styles.css` | UI без стилей, но работает | `load_css` использует встроенные default-стили |
| Передать в `cv2.VideoCapture` отсутствующее устройство | Предупреждение `Не удалось открыть веб-камеру`, `webcam_running = False` | Проверка `cap.isOpened()` |
| Закрыть камеру через ОС во время работы | Исключение в `cap.read()`, цикл прерывается, finally-блок отрабатывает | Нет автоматического восстановления: нужно вручную применить "⏹️ Стоп" + "▶️ Запустить" |

### Обязательные действия

- Перезагружать страницу Streamlit после изменения параметров пайплайна и перезапуска backend.
- Закрывать камеру явно перед закрытием вкладки.
- Перезапускать utility после смены модели (`enet_b0_8` ↔ `enet_b2_8`) – модель загружается один раз.
